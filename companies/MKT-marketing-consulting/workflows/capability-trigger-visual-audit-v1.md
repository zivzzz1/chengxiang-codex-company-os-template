# 能力触发可视审计 V1

本工作流用于锁定对话里的能力触发、过程变化和完成前核销。它管的是回话方法，不是生成一个额外文件。

## 触发条件

命中任一条件时触发：

1. 用户问“触发什么能力 / 用了什么能力 / 能力审计 / 计划里要看到能力”。
2. 新任务、项目推进、系统建设、客户可见材料、视觉、PPT、HTML、图片、视频、报价、合同、同步或发布。
3. 完成前需要说“已完成 / 可交付 / 已固化 / 已同步”。

极小单点请求可以简化为：

```text
能力触发：无新增触发
```

## 输出顺序

重要任务的回话顺序固定为：

1. 任务定型表。
2. 执行前置声明。
3. 技能路由锁定。
4. 能力触发预判面板。
5. 正式推进。
6. 完成审计表。

## 技能路由锁定

```text
技能路由锁定：
- 用户点名技能：<用户明确点名的 Skill / 能力 / 部门；没有写无>
- 自动命中技能 / 能力：<由任务类型、交付物、风险 Gate 自动命中的能力>
- 本轮明确不触发：<容易误触发但本轮不该用的能力>
- 完成前必须核销：<交付前必须逐项确认的技能调用结果>
```

规则：

1. 用户点名的技能必须逐项列名，不能写成“按要求调用技能”。
2. HTML 汇报命中 `html-visual-reporting`。
3. 一图看懂 / 信息图 / 结构图命中 `one-image-understanding`。
4. 图片最终交付命中生图模型硬 Gate。
5. 字体 / 版式 / 客户可见视觉命中字体审美检查。
6. 项目进度汇报命中项目恢复和能力触发可视审计。
7. 锁定后的技能必须改变执行路径、产出结构或验收项；事后补说明不算触发。

## 能力触发预判面板

| 能力 | 是否触发 | 为什么触发 | 负责什么 | 不负责什么 |
| --- | --- | --- | --- | --- |
| <能力名> | 已触发 / 建议触发 / 不触发 | <触发原因> | <本轮职责> | <边界> |

只列本轮相关能力，不展开全量能力表。

## 过程状态条

执行中出现能力变化、强制 Gate、漏触发、用户改主线或执行失败时，输出：

```text
能力触发：
- 已触发：<能力 A>、<能力 B>
- 建议触发：<能力 C>
- 未触发：<能力 D>
- 原因：<一句话说明>
```

## 完成审计表

完成、封存、同步、发布或声明可用前必须输出：

| 审计项 | 结论 |
| --- | --- |
| 计划中应触发能力 | <能力列表> |
| 实际触发能力 | <能力列表> |
| 技能路由锁定核销 | <用户点名技能 / 自动命中能力是否逐项核销> |
| 未触发但需要解释 | <没有则写无> |
| 是否越权 | 是 / 否；如是，写明边界 |
| 是否有能力遗漏 | 是 / 否；如是，写明补救 |
| 用户需要判断 | <用户要确认的点> |

只要发现点名技能未触发、边界越权或验证证据不足，不能说“已完成”，只能说“阻塞 / 待确认 / 已完成部分”。
